# Стандартен Контракт за Извличане на Данни от Платежни Документи (JSON Schema Proposal)

Този документ дефинира унифицирана JSON структура (контракт), която трябва да бъде стриктно спазвана при извличане на структурирана информация от неструктурирани платежни документи (снимки на фактури, касови бележки, известия). 

Целта на контракта е да служи като единен стандарт за оценка и бенчмарк на различни AI модели (Gemini, Claude, GPT/Codex) върху един и същ набор от тестови изображения.

## 1. Примерна JSON Структура

Моделите трябва да връщат **САМО валиден JSON**, отговарящ на следната структура:

```json
{
  "metadata": {
    "document_category": "invoice", 
    "language": "bg",
    "confidence_level": "high", 
    "needs_human_review": false,
    "unreadable_parts_found": false
  },
  "parties": {
    "issuer": {
      "name": "ИМЕ НА ФИРМАТА ДОСТАВЧИК",
      "company_id": "ЕИК / Булстат", 
      "vat_id": "ДДС Номер (напр. BG123456789)",
      "address": "Пълен адрес на управление/обект",
      "iban": "IBAN сметка (ако е налична)"
    },
    "recipient": {
      "name": "ИМЕ НА ФИРМАТА ПОЛУЧАТЕЛ",
      "company_id": "ЕИК / Булстат",
      "vat_id": "ДДС Номер",
      "address": "Адрес на получателя"
    }
  },
  "document_details": {
    "document_number": "Номер на документа (напр. 0000000123)",
    "issue_date": "YYYY-MM-DD",
    "due_date": "YYYY-MM-DD (ако е посочена)",
    "payment_method": "cash | bank_transfer | card | other", 
    "currency": "Валута (напр. BGN, EUR)"
  },
  "financials": {
    "total_net_amount": 100.00,
    "total_tax_amount": 20.00,
    "total_gross_amount": 120.00,
    "tax_breakdown": [
      {
        "tax_rate_percent": 20,
        "tax_base_amount": 100.00,
        "tax_amount": 20.00
      }
    ]
  },
  "line_items": [
    {
      "row_number": 1,
      "description": "Описание на закупената стока/услуга",
      "quantity": 2.0,
      "unit_measure": "бр.",
      "unit_price_net": 50.00,
      "total_net_amount": 100.00,
      "tax_rate_percent": 20
    }
  ],
  "validation": {
    "is_math_correct": true,
    "math_discrepancy_details": "Детайли, ако сметките не излизат",
    "missing_critical_fields": [],
    "warnings": []
  }
}
```

## 2. Описание на логическите блокове

### 2.1 Metadata (Метаданни)
Описва състоянието на самия процес по разпознаване.
* **document_category**: `invoice`, `receipt`, `credit_note`, `debit_note`, `other`.
* **confidence_level**: `high`, `medium`, `low`. Позволява на системата да прецени колко да се довери на резултата.
* **needs_human_review**: Булев флаг. Вдига се, когато моделът изпитва затруднения или открие несъответствия.

### 2.2 Parties (Страни по сделката)
Ясно разделение между Издател (Issuer) и Получател (Recipient). 
* **Забележка**: При касови бележки (`receipt`), блокът `recipient` обикновено ще съдържа `null` стойности, освен ако бележката не съдържа данни за купувача (напр. фактура от касов апарат).

### 2.3 Financials (Финансови данни)
Обобщени финансови резултати за целия документ.
* **tax_breakdown**: Изключително важен масив за правилното осчетоводяване. Документите могат да съдържат смесени данъчни ставки (напр. 20% ДДС и 9% ДДС). Масивът трябва да групира данъчната основа и данъка за всяка отделна ставка.

### 2.4 Validation (Самооценка и Валидация)
Задължава модела да провери собствения си резултат:
* **is_math_correct**: Моделът трябва да калкулира:
  1. `Сумата от total_net_amount на всички line_items` == `financials.total_net_amount`
  2. `financials.total_net_amount + financials.total_tax_amount` == `financials.total_gross_amount`
* **missing_critical_fields**: Списък (масив от стрингове), съдържащ пътеки към ключови полета, които моделът не е успял да намери (напр. `["document_details.issue_date", "parties.issuer.company_id"]`).

## 3. Инструкция към AI Моделите
Ако подавате този контракт към друг модел, използвайте следния системен промпт (System Prompt):

> Ти си експертна система за OCR и извличане на структурирани данни. Ще ти бъдат подавани изображения на финансови документи. Твоята задача е да извлечеш информацията от изображението и да върнеш **ЕДИНСТВЕНО И САМО** валиден JSON, който стриктно спазва подадената схема (Контракт). 
> Ако някои данни липсват от документа, задай стойност `null`, но не премахвай ключа от JSON структурата. 
> Преди да генерираш крайния JSON, изчисли математическата коректност в блока `validation`.
